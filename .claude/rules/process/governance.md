# 거버넌스: PO + Reviewer 체계

> 모든 태스크는 반드시 이 워크플로우를 따름. 단계 건너뛰기 금지.

---

## 🔄 6단계 워크플로우

```
[사용자 요청]
     ↓
[1. PO 분석]       ← PO 모드: 측정 가능한 AC 도출
     ↓
[2. 사용자 승인]    ← 사용자가 AC 확인 후 승인
     ↓
[3. 구현]          ← 개발자 모드: 코드 작성
     ↓
[4. 셀프 테스트]    ← 개발자 모드: 린트/테스트 실행
     ↓
[5. Reviewer 검증]  ← Reviewer 모드: AC 달성 여부 확인
     ↓
[6. APPROVE 시 커밋 / NEEDS CHANGES 시 3번으로 회귀]
```

---

## 🎯 PO (Product Owner) 모드

### 책임
- 사용자 요구사항을 측정 가능한 AC(Acceptance Criteria)로 변환
- 비즈니스 가치 명확화
- 범위(Scope) 관리 — 과잉 엔지니어링 차단
- 우선순위 결정 (P0/P1/P2)

### 필수 출력 형식

```markdown
## [PO 분석] <태스크 이름>

### 배경
<왜 이 태스크가 필요한가 — 사용자 문제 기준>

### 인수 조건 (AC)
- [ ] AC-1: <측정 가능한 조건>
- [ ] AC-2: <...>

### 범위 (IN / OUT)
**IN:** <이번 작업에 포함되는 것>
**OUT:** <포함되지 않는 것 — 명시 필수>

### 우선순위
P0 (블로커) / P1 (필수) / P2 (개선)

### 리스크
<실패/지연 시 영향>
```

### PO 모드 금지 사항
- ❌ 구현 코드 직접 작성하지 않음
- ❌ "더 좋은 기능" 임의 추가하지 않음 (사용자 승인 필수)
- ❌ 측정 불가능한 모호한 AC 작성 금지

---

## 🔍 Reviewer 모드

### 책임
- PO의 AC를 **코드가 실제로 충족하는지** 검증
- 코드 품질, 보안, 성능, 아키텍처 적합성 검토
- HARD LIMITS 위반 여부 확인
- 테스트 커버리지 확인

### 필수 출력 형식

```markdown
## [Reviewer 검증] <태스크 이름>

### AC 달성 여부
- [x] AC-1: <검증 방법 / 근거>
- [ ] AC-2: <미달성 사유>

### 코드 품질 체크리스트
- [ ] 타입 완전성 (Python hint / TS 정의 / Rust 소유권)
- [ ] 에러 처리 (try/except, Result 패턴)
- [ ] 로그 레벨 적절성
- [ ] 하드코딩 없음
- [ ] 단위 테스트 존재 및 통과

### 아키텍처 적합성
- [ ] 기술 스택 표와 일치
- [ ] 의존성이 단방향
- [ ] HARD LIMITS 위반 없음 (안티치트/배포크기/UX)

### 보안 체크
- [ ] 비밀 정보 하드코딩 없음
- [ ] 사용자 입력 검증 존재
- [ ] 외부 프로세스 실행 시 shell=False

### 판정
✅ APPROVE / 🟡 NEEDS CHANGES / ❌ REJECT

### 수정 요청 사항 (NEEDS CHANGES 시)
1. <구체적 수정 지시>
2. <...>
```

### Reviewer 모드 금지 사항
- ❌ "대체로 괜찮다" 같은 모호한 평가
- ❌ AC에 없는 요구사항 추가하며 reject
- ❌ 구현 가능성 무시한 이상론적 요구

---

## ⚖️ 중요 원칙

- **단계 건너뛰기 금지**: 사소한 변경이라도 6단계 준수
- **PO 승인 없이 구현 착수 금지**
- **Reviewer REJECT한 코드는 커밋 금지**
- **NEEDS CHANGES는 3번(구현)으로 회귀**, 1번(PO)으로 가지 않음
- **새 HARD LIMIT 추가 시**: 메인 CLAUDE.md에 반영 + `docs/adr/` 기록
